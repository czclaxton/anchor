**Actionable findings**

**Finding 1 — `src/lib/progress.ts` lines 10, 23, 31: No JSON.parse error handling (AS1, diff-derived AS3)**

All three exported functions call `JSON.parse(existing)` without a try/catch. A `SyntaxError` from malformed or externally injected localStorage data propagates uncaught into the React island's `useEffect`, crashing the component with no recovery. This breaks finding-discipline clause 2: a user who has corrupt data in any `anchor:progress:*` key loses all progress UI functionality permanently until they manually clear localStorage.

Fix: wrap each `JSON.parse` call in try/catch, returning a safe default on failure:

```typescript
function parseSlugArray(raw: string): string[] {
  try {
    const parsed = JSON.parse(raw)
    return Array.isArray(parsed) ? parsed : []
  } catch {
    return []
  }
}
```

Replace all three inline `JSON.parse(existing) as string[]` calls with `parseSlugArray(existing)`. This also resolves Finding 2 below.

**Finding 2 — `src/lib/progress.ts` lines 10, 23, 31: Unchecked array cast after parse (AS2)**

`JSON.parse(existing) as string[]` is a TypeScript-only assertion. At runtime, if the stored value is a JSON number, boolean, or object, `.includes()` in `isComplete` and `.length` in `getProgress` throw `TypeError: slugs.includes is not a function`. Finding-discipline clause 2 applies: valid JSON that is not an array produces a runtime crash, not a safe fallback. Resolved by the `Array.isArray` guard in the `parseSlugArray` helper above.

**Finding 3 — `src/components/AnchoredButton.tsx` line 37: `animate-anchor-drop` class undefined (diff-derived AS7)**

**Advisory Observations**

- **A1 — Multi-tab write race (diff-derived AS5):** Two tabs completing different lessons simultaneously each read a stale empty array and the last writer's lesson overwrites the first. This is an inherent localStorage limitation; the deduplication guard in `markComplete` only prevents same-lesson duplicates within one tab. No locking mechanism exists. This is data loss on a real concurrent path but is not in the ticket's stated scope and localStorage has no compare-and-swap primitive. A StorageEvent listener (which fires in _other_ tabs) could partially mitigate display staleness but not the write race itself.
