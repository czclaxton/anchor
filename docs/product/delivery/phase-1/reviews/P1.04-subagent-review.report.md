**Actionable findings**

**Finding 1 — `builtins.input` mutation persists indefinitely across `runPython` calls**
File: `src/lib/runPython.ts`, lines 37–48.
After any call to `runPython` with `onInput`, the Python builtin `builtins.input` is permanently replaced with `_async_input` in the Pyodide singleton. The `else` branch (no `onInput`) never restores the original. Any subsequent call without `onInput` whose code contains `input()` will silently invoke the stale `_async_input` coroutine and attempt to call whatever `_onInput` JS function was last registered, routing interactive prompts to the wrong handler, hanging indefinitely, or raising a Python `NameError`. Discipline 2 applies: a demonstrable correctness gap.
Recommended fix: Save the original `builtins.input` before replacing it, then restore it after `runPythonAsync` completes in the `onInput` branch. Concretely, in the Python setup string, save `_original_input = builtins.input`, and add a `finally` block or a post-execution restoration:
```python
builtins.input = _original_input
```
Alternatively, restore it in JS after the `onInput` branch's `runPythonAsync` resolves:
```javascript
await pyodide.runPythonAsync('import builtins; builtins.input = _original_input')
```
where `_original_input` was stashed before the first mutation.

**Finding 2 — Pyodide load failure creates a permanently unrecoverable stuck state**
File: `src/lib/runPython.ts`, lines 10–19.
If `loadPyodide()` rejects (WASM 404, network timeout), the IIFE becomes a rejected Promise stored in `pyodideLoading`. Because `getPyodide()` checks `if (pyodideLoading) return pyodideLoading` before attempting a new load, all future calls return the same rejected Promise immediately — no retry is possible without a full page reload. Discipline 3 (spec-permits-real-bug): the ticket spec requires lazy loading but does not specify retry behavior; the implementation makes transient network failures permanent.
Recommended fix: In the IIFE, add a catch that resets `pyodideLoading = null` so the next `getPyodide()` call re-enters the loading path:
```javascript
pyodideLoading = (async () => {
  const { loadPyodide } = await import('pyodide')
  pyodideInstance = await loadPyodide()
  return pyodideInstance
})().catch((err) => {
  pyodideLoading = null
  throw err
})
```

**Finding 3 — Unmounted component leaves Pyodide permanently suspended**
File: `src/components/PythonEditor.tsx`, lines 46–57 (`handleRun`).
If the component unmounts while `runPython` is suspended on an `input()` Promise (the user navigates away mid-prompt), the `onInput` callback's `resolve` is captured inside a stale `inputState` that React has discarded. `setInputState` on an unmounted component is silently ignored; the input form never renders; `handleInputSubmit` is never called; the Promise never resolves. Pyodide's `runPythonAsync` is permanently suspended. Because Pyodide is single-threaded, this blocks all future Python executions on the singleton — any re-mounted `PythonEditor` will hang. Discipline 3 (spec-permits-real-bug): the spec does not address unmount cleanup.
Recommended fix: In the `onInput` callback, track a module-level or ref-backed cancellation token. On `useEffect` cleanup (unmount), set the token so the next tick resolves the pending Promise with a rejection (e.g. `new Error('cancelled')`), which propagates through `_async_input` as a Python exception and unblocks `runPythonAsync`. Example: use a `useRef<(() => void) | null>(null)` that stores the `resolve`/`reject` pair and is called with a rejection in the `useEffect` cleanup.

**Advisory Observations**

- **A1 — Regex matches `input(` inside Python string literals.** The `/\binput\s*\(/g` transform in `wrapWithAsyncInput` (`runPython.ts:54`) operates on raw source text. Code like `msg = "call input(x)"` is transformed to `msg = "call await input(x)"`, silently changing the string content. Similarly, `eval("input('x')")` becomes `eval("await input('x')")`, which would raise a Python `SyntaxError` inside `eval`. For an educational editor at this stage the risk is low, but it's a hidden limitation worth noting as the editor matures.

- **A2 — Rationale section never filled in (doc drift).** `docs/product/delivery/phase-1/ticket-04-python-editor.md` lines 53–56 contain only unfilled placeholders (`Red first: [what test failed first]`, `Why this path: [...]`, `Alternative considered: [...]`, `Deferred: [...]`). These fields document implementation decisions for future readers and the retrospective. The ticket was delivered without completing this section.

- **A3 — Green section describes `sys.stdin` override; implementation overrides `builtins.input` instead.** The ticket's Green section (line 33) states: "Intercept `input()` by overriding `sys.stdin` in Pyodide's Python environment with a JS-backed async reader." The actual implementation replaces `builtins.input` directly (`runPython.ts:43`). These are different mechanisms with different behavioral scope. The doc describes the alternative approach that was not taken; the rationale for choosing `builtins.input` over `sys.stdin` is not recorded.

- **A4 — No automated test for multiple sequential `input()` calls.** The ticket's Review Focus explicitly flags "run a multi-input script (two `input()` calls) and confirm each one pauses and resumes correctly in sequence" as a verification step. This is only covered by manual testing, not by the Vitest suite. A test with `x = input('a?')\ny = input('b?')\nprint(x, y)` and two `mockResolvedValueOnce` calls would close this gap.

- **A5 — `setStdout` is not concurrency-safe at the library level.** `runPython` calls `pyodide.setStdout` on line 32, replacing whatever handler was installed by a previous in-flight call. The `PythonEditor` component guards against this via `isRunning`, but `runPython` is exported as a public API with no internal guard. Concurrent programmatic calls will silently corrupt each other's stdout capture. A code comment on `runPython` noting "not safe for concurrent invocations" would communicate this constraint to future callers without requiring a change to the implementation.
