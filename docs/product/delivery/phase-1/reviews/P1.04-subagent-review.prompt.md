### Invariants to hold

1. `runPython("print('hello')")` must resolve with `{ stdout: "hello\n" }` — stdout is captured from Pyodide's Python output and returned in full.
2. `runPython("x = input('name?')\nprint(x)", mockInput)` must call `mockInput("name?")`, await the resolved value, and return `{ stdout: "Alice\n" }` when mock resolves with "Alice" — the `input()` interception bridge must bridge async JS back to synchronous Python execution.
3. The Pyodide singleton (`pyodideInstance`) must be loaded at most once per JS runtime — concurrent `runPython` calls must share the same instance and not double-load WASM.

### Attack surfaces to probe

1. **`wrapWithAsyncInput` regex transform** — the regex `/\binput\s*\(/g` replaces `input(` globally in user code. Does this correctly transform multi-line code, code with `await input(` already present (double-await), or code using `input` as a variable name (e.g. `user_input = 5`)?
2. **Python async bridge via `_onInput` global** — `pyodide.globals.set('_onInput', ...)` is called before `wrapWithAsyncInput` runs the modified code. If `runPython` is called concurrently for two snippets that both use `input()`, can `_onInput` be overwritten mid-execution, causing one call to use the wrong handler?
3. **Pyodide singleton race condition** — `getPyodide()` sets `pyodideLoading` then returns it. If two callers hit `getPyodide()` before `pyodideInstance` is set (both see `null`), does the implementation correctly share the single loading Promise or create two concurrent `loadPyodide()` calls?
4. **`builtins.input` mutation persistence** — `runPython` replaces `builtins.input` with `_async_input` when `onInput` is provided. If a subsequent call to `runPython` without `onInput` runs `print(input("name?"))`, will it use the mutated async `input()` and hang, since there is no handler?
5. **`handleRun` Promise leak on component unmount** — if the user unmounts the component (navigates away) while `runPython` is awaiting an `onInput` Promise, the Promise resolves on a stale `setInputState`/`setOutput` reference, potentially calling React state setters on an unmounted component.
6. **`stdout` accumulation with `setStdout` batched** — `setStdout` is called on every `runPython` invocation. If Pyodide's `setStdout` replaces the previous handler, is the `stdout` variable in the closure still fresh (initialized to `''`) for each call, or can it accumulate across calls due to closure capture over a module-level variable?
